# ⚙️ Техническая спецификация (TechSpec)

## 📌 Технический обзор

> **Мультиплатформенное приложение** для безопасного совместного доступа к папкам и файлам **без хранения файлов на сервере**.
> Централизованное хранение **только метаданных** в PostgreSQL, разграничение прав доступа, генерация защищённых ссылок, интеграция с Проводником Windows, поддержка Android и Web-клиентов, email и внутренние уведомления, расширенный поиск и административное управление.

Архитектура построена на **централизованном REST API** и реляционной БД **PostgreSQL**.

---

## 🧱 Системный дизайн

### 🔧 Backend

- **Центральный REST API** для всех клиентов (Web, Android, Windows)
- **PostgreSQL** – хранение метаданных (пользователи, папки, файлы, права, ссылки, события)
- **Аутентификация** – OAuth 2.0 (парольный поток для внутренних клиентов)
- **Безопасность** – все действия через **HTTPS**
- **Управление сессиями** и токенами доступа
- **Генерация криптоустойчивых токенов** для share‑ссылок, хранение только хэшей
- **Проверка ссылок** – срок действия (expires_at) и статус отзыва (revoked)
- **Права доступа** – `view` / `full`, проверка владельца на каждом запросе
- **Ручной отзыв ссылок** – мгновенное прекращение доступа
- **Административные эндпоинты** – управление пользователями, правами и метаданными
- **Внутренние события** – для уведомлений (создание доступа, отзыв, изменение прав)
- **Webhooks** – внутренние push‑уведомления с подтверждением доставки и повторной отправкой
- **Email‑уведомления** – через внешний SMTP-сервер
- **Ошибки** – возвращаются с соответствующим статусом и сообщением (истёкшая/отозванная ссылка, доступ к чужим данным и т.д.)
- **Нет хранения файлов** – только метаданные и ссылки на локальные файлы
- **signaling-service** – отдельный микросервис, сводит WebRTC-пиры (SignalR/WebSocket). Никаких файлов через него не проходит — только обмен SDP-offer/answer и ICE-кандидатов.

### 🔀 P2P-транспорт файлов (WebRTC / ICE / DTLS)

**Ключевой инвариант:** байты файла **никогда** не проходят через прикладные сервера. Передача идёт напрямую между клиентами по WebRTC data channel. Сервер нужен только для знакомства пиров и, при худшем сетевом раскладе, для ретрансляции зашифрованных пакетов через TURN.

**Слои стека:**

| Слой | Роль | Реализация |
|---|---|---|
| **signaling** | сводит пиров, прокидывает SDP + ICE-кандидатов | `signaling-service` (ASP.NET Core + SignalR), авторизация JWT от auth-service + проверка прав/share-токена |
| **STUN** | определение внешнего адреса пира (srflx-кандидат) | публичные серверы (Google, Cloudflare), без self-hosting |
| **TURN** | fallback-ретранслятор для симметричных NAT | собственный **coturn**, временные креды через REST-API (RFC 8489), выдаёт signaling-service |
| **ICE** | перебор host → srflx → relay и выбор рабочей пары | встроен в `RTCPeerConnection` (браузер / SIPSorcery) |
| **DTLS** | шифрование data-channel на транспортном уровне | обязательное, AES-GCM; встроено в WebRTC |
| **App-level E2E** | шифрование самого файла на клиенте | AES-GCM, обмен ключами по ECDH внутри data channel до начала передачи |

**Зачем двойной слой шифрования.** DTLS закрывает канал между пирами — перехватчик и оператор TURN видят только случайные байты. Но оператор TURN знает метаданные (кто-с-кем, когда, сколько). Поэтому поверх DTLS сам файл дополнительно шифруется на клиенте — ключ известен только пирам, сервер не может расшифровать даже теоретически.

**C#-стек:** библиотека **SIPSorcery** — `RTCPeerConnection`, data channels, встроенный ICE-агент. Используется в Windows-клиенте и интеграционных тестах. Web-клиент — нативный `RTCPeerConnection` браузера. Android — `google-webrtc`.

**Состояние signaling-сервиса эфемерно** и хранится в Redis (список активных комнат/пиров, TURN-креды с TTL). В PostgreSQL signaling ничего не пишет.

### 📱 Frontend (Web, Android, Windows)

- Вход и регистрация через email/пароль (OAuth 2.0)
- Просмотр, создание, удаление папок и файлов, загрузка файлов (метаданные)
- Интерфейс назначения прав доступа зарегистрированным пользователям
- Генерация и управление ссылками с указанием срока действия
- Вкладка **«Доступные папки»** – сортировка и фильтрация по владельцу, названию, правам
- Расширенный поиск по названию, типу, владельцу, правам
- Уведомления (email и внутренние push через webhooks)
- Обработка ошибок соединения, авторизации, доступа, загрузки/удаления
- **Нет кэширования данных других пользователей**
- Интерфейс администратора для управления пользователями и правами
- Соответствие UI стандартам каждой платформы

### 🪟 Интеграция с Проводником Windows

- Через **Windows Shell Extensions** (или Cloud Files API — см. roadmap, точка решения)
- Папки и файлы пользователя отображаются как **виртуальная папка**
- **Метаданные** синхронизируются с сервером по запросу
- **Содержимое файлов** скачивается/заливается напрямую с пира-владельца через WebRTC (см. раздел "P2P-транспорт")
- **Нет локального кэширования** данных других пользователей
- Обработка ошибок соединения и синхронизации с отображением уведомлений

### 🗄️ Данные и хранилище

- Вся метаинформация – в **PostgreSQL**
- Только **хэши токенов** для share‑ссылок
- **Нет хранения файлов** – только ссылки на локальные файлы
- **Эфемерное состояние signaling-service** (активные пиры, TURN-креды с TTL) — в **Redis**
- **Жёсткое разграничение доступа** на уровне БД и API

### 🔔 Уведомления

- **Email** – через внешний SMTP (SendGrid, Mailgun, Amazon SES и др.)
- **Внутренние push** – через webhooks с подтверждением доставки
- **Повторная отправка** при отсутствии подтверждения
- Типы уведомлений: предоставление доступа, отзыв, потеря доступа, ошибки

### 🛡️ Безопасность

- **Только HTTPS** для сигнального/REST-трафика
- **OAuth 2.0** + управление сессиями
- Проверка прав на **каждом запросе**
- Криптостойкие токены для ссылок
- Мгновенное прекращение доступа при отзыве прав или ссылки
- Файлы **не хранятся** на сервере
- **DTLS** обязательно шифрует WebRTC data-channel между пирами
- **App-level E2E** — сам файл дополнительно шифруется на клиенте (AES-GCM); ключ обменивается по ECDH между пирами и сервер его не знает
- **TURN** видит только зашифрованные пакеты; метаданные (кто-кому-сколько) логируются минимально, с TTL

### 🧪 Тестирование

- Юнит-тесты бизнес-логики
- Интеграционные тесты REST API и БД
- E2E тесты основных сценариев на всех платформах
- Тестирование обработки ошибок и edge‑cases (истёкшие/отозванные ссылки, потеря соединения, доступ к чужим данным)

## 📊 Модель данных / Схема БД

| Таблица       | Колонка         | Тип                 | Описание |
|---------------|----------------|---------------------|----------|
| **users**     | id              | UUID (PK)           | Уникальный ID пользователя |
|               | email           | VARCHAR, unique     | Email пользователя |
|               | password_hash   | VARCHAR             | Хэш пароля |
|               | is_admin        | BOOLEAN             | Признак администратора |
| **folders**   | id              | UUID (PK)           | Уникальный ID папки |
|               | owner_id        | UUID (FK → users)   | Владелец папки |
|               | name            | VARCHAR             | Название папки |
| **files**     | id              | UUID (PK)           | Уникальный ID файла |
|               | folder_id       | UUID (FK → folders) | Родительская папка |
|               | owner_id        | UUID (FK → users)   | Владелец файла |
|               | name            | VARCHAR             | Название файла |
|               | type            | VARCHAR             | Тип файла |
| **permissions** | id            | UUID (PK)           | Уникальный ID права |
|               | folder_id       | UUID (FK → folders) | Папка |
|               | user_id         | UUID (FK → users)   | Пользователь |
|               | access_level    | ENUM (`view`, `full`) | Уровень доступа |
| **share_links** | id            | UUID (PK)           | Уникальный ID ссылки |
|               | folder_id       | UUID (FK → folders) | Папка |
|               | token_hash      | VARCHAR             | Хэш токена |
|               | expires_at      | TIMESTAMP           | Срок действия |
|               | revoked         | BOOLEAN             | Признак ручного отзыва |
| **events**    | id              | UUID (PK)           | Уникальный ID события |
|               | user_id         | UUID (FK → users)   | Получатель уведомления |
|               | type            | VARCHAR             | Тип события (`access_granted`, `access_revoked` и др.) |
|               | data            | JSONB               | Дополнительные данные |
|               | delivered       | BOOLEAN             | Подтверждение доставки |
|               | created_at      | TIMESTAMP           | Время создания |

## 🔗 Связанные компоненты (репозитории)

- **backend-rest-api** – центральный REST API, бизнес-логика, интеграция с БД
- **signaling-service** – WebRTC signaling (SignalR hub, выдача TURN-кредов)
- **turn-infra** – развёрнутый coturn, публичные STUN берём как есть
- **web-client-react** – веб-интерфейс (React) с нативным WebRTC P2P-слоем
- **android-client** – Android-приложение (`google-webrtc`)
- **windows-client** – Windows-клиент + Shell Extension + SIPSorcery для P2P
- **db-migrations** – миграции и схемы PostgreSQL
- **notifications-service** – обработка email и webhook-уведомлений
